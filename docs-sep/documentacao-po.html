O produto SEP é uma plataforma que conecta empresas tomadoras de empréstimo a empresas credoras, com operação assistida por um time financeiro interno e governança do administrador do sistema. A fase atual entrega a fundação técnica da API, preparando o terreno para as próximas ondas funcionais: onboarding, análise de crédito, formalização contratual e cobrança.

Importante: esta fase não entrega o produto completo. Ela constrói a base segura, auditável e documentada sobre a qual as jornadas de negócio serão construídas em ondas.

Decomposição dos épicos ativos em histórias de usuário organizadas por valor de negócio.

1Fundação da API

Base técnica que sustenta todas as jornadas futuras.

  • Configurar Docker Compose com PostgreSQL local
  • Configurar locale pt-BR e timezone America/Sao_Paulo
  • Implementar auditoria JPA (criação e modificação)
  • Padronizar identificadores UUID em todas as entidades
  • Configurar Flyway para versionamento do banco
  • Expor endpoint de healthcheck via Actuator
  • Prever CORS para integração futura com Angular
Fase atual

2Gestão de Usuários (Sprint 2)

Base de identidade da plataforma — criação de contas.

  • Como visitante, posso criar um usuário sem autenticação
  • Os dados do usuário ficam persistidos com identificador estável (UUID)
  • Auditoria registra quem criou e quando, com fallback técnico quando não há login
  • Minha senha fica armazenada de forma segura (hash BCrypt)
  • Em nenhuma resposta da API a senha aparece
Fase atual

3Segurança, Autenticação e Autorização (Sprint 3)

Controle de acesso seguro, auditável e com ownership.

  • Como usuário, posso autenticar com e-mail e senha
  • O sistema emite um JWT com identificador e perfil do usuário
  • Como usuário autenticado, posso consultar meus dados em /me
  • Como admin, posso consultar qualquer usuário pelo id
  • Como cliente, posso consultar apenas os meus dados
  • Como admin, posso listar todos os usuários
  • Como usuário autenticado, posso alterar minha própria senha
  • Autorização respeita perfil e ownership do recurso
Fase atual

4Tratamento de Erros, Documentação e Testes (Sprint 4)

Contratos estáveis e qualidade comprovada por testes.

  • Todos os erros da API seguem um payload padronizado
  • A documentação OpenAPI fica disponível via Swagger UI
  • Cada endpoint possui exemplos e schemas explícitos
  • Os códigos de resposta seguem convenção REST
  • Os cenários críticos de autenticação e autorização são cobertos por testes automatizados
Fase atual

5Onboarding KYC/KYB

Validação cadastral e documental de pessoas e empresas.

  • Coleta e validação de documentos obrigatórios
  • Verificação de identidade e dados cadastrais
  • Trilha operacional de aprovação, rejeição e pendências
  • Preparação para integrações futuras (OCR, biometria)
Próxima onda

6Análise de Crédito

Esteira de decisão de risco da proposta.

  • Preparação para assinatura eletrônica e CCB
Próxima onda

7. Arquitetura em Alto Nível

Visão simplificada de como o produto se organiza.
Frontend Angular consome a API SEP via REST. A API grava em PostgreSQL. A integração com Celcoin BaaS entra em fase posterior.
Arquitetura inicial: monolito modular. Microsserviços não fazem parte da fase inicial — essa decisão reduz risco e complexidade até a estabilização do fluxo de contratação.

Fronteiras entre Épicos

Para evitar sobreposição de responsabilidades, as fronteiras entre épicos foram explicitadas no PRD: onboarding não substitui a jornada da credora; análise de crédito não substitui o backoffice; formalização não substitui a análise; cobrança não substitui o meio de pagamento; Pix não substitui cobrança, análise ou formalização; frontend não concentra regra de negócio de domínio.
