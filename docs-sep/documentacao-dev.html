2. Arquitetura do Backend

Monolito modular com camadas explícitas. Microsserviços ficam fora da fase atual.
Fluxo vertical: cliente → controller → security → service → repository → banco.
Regra das camadas: controllers não implementam regra de negócio, DTOs não são reutilizados como entidades JPA, classes de segurança não carregam regra de domínio, services não acessam HTTP diretamente.

3. Estrutura de Pacotes

Separação clara por responsabilidade desde o início.
com.dynamis.broker_app

├── security                  # JwtTokenProvider, JwtAuthenticationFilter,
│                             # CustomUserDetailsService
├── exception                 # ApiExceptionHandler, exceções de domínio
├── model                     # Usuario (entidade JPA)
├── repository                # UsuarioRepository

└── web
    ├── controller            # AuthController, UsuarioController
    └── dto                   # UsuarioCreateDto, LoginRequestDto, ...
        └── mapper            # UsuarioMapper

Subdivisões quando o domínio crescer

À medida que novas épicas entrarem (onboarding, crédito, formalização, cobrança), recomenda-se subdividir por agregado: service.auth, service.usuario, service.onboarding, repository.usuario, web.controller.auth, web.controller.usuario, etc.

5. Modelo de Domínio e DTOs

Entidade principal e contratos de DTO da fase atual.

Entidade Usuario

@Entity
@Table(name = "usuario")
@EntityListeners(AuditingEntityListener.class)
public class Usuario {

    @Id
    @Column(name = "id", columnDefinition = "uuid")
    private UUID id;                    // UUID v6

    @Column(name = "username", unique = true, nullable = false)
    private String username;              // e-mail

    @Column(name = "password", nullable = false)
    private String password;              // BCrypt hash

    @Enumerated(EnumType.STRING)
    @Column(name = "role", nullable = false)
    private Role role;                   // ADMIN | CLIENTE

    @CreatedDate       private OffsetDateTime dataCriacao;
    @LastModifiedDate  private OffsetDateTime dataModificacao;
    @CreatedBy         private String criadoPor;      // UUID ou "system"
    @LastModifiedBy    private String modificadoPor;  // UUID ou "system"
}

DTOs

UsuarioCreateDto

Cadastro público · sem autenticação.

Resposta pública · nunca expõe senha.

  "role": "ADMIN",
  "dataCriacao": "2026-04-24T18:30:00-03:00",
  "dataModificacao": "2026-04-24T18:30:00-03:00",

  "modificadoPor": "system"

Troca da própria senha.

6. Endpoints da Fase Atual

Todos em /api/v1/*. Datas em ISO-8601 com offset.
Método Path Acesso Finalidade
POST /api/v1/usuarios público Criar usuário (ADMIN ou CLIENTE)
GET /api/v1/usuarios/{id} ADMIN (qualquer) · CLIENTE (próprio) Consultar usuário por id
GET /api/v1/usuarios ADMIN Listar todos os usuários
PATCH /api/v1/usuarios/{id}/senha próprio usuário autenticado Alterar a própria senha
POST /api/v1/auth/login público Autenticar e receber JWT
GET /api/v1/auth/me autenticado Retornar o usuário do token

7. Segurança · JWT · Auditoria

Estratégia de autenticação, autorização e rastreabilidade.
JWT emitido: header + claims mínimas + assinatura HMAC. Sem refresh token nesta fase.

Auditoria JPA

  • Campos obrigatórios em toda entidade relevante: dataCriacao, dataModificacao, criadoPor, modificadoPor.
  • criadoPor e modificadoPor priorizam o UUID do usuário autenticado.
  • Fallback técnico quando não há autenticação: valor literal "system".
  • O AuditorAware<String> deve extrair o sub do token a partir do SecurityContextHolder.
  • Sem soft delete nesta fase — exclusões são físicas ou evitadas por regra de negócio.

Autorização

  • Perfis disponíveis: ROLE_ADMIN e ROLE_CLIENTE.
  • Admin: consulta/lista qualquer usuário.
  • Cliente: consulta apenas o próprio (ownership validado no service comparando principal.id com pathVariable id).
  • Cliente recebe 403 Forbidden ao tentar listar.

8.1 Roadmap Técnico · Épicos do Projeto

Visão consolidada de todas as grandes implementações previstas para a plataforma SEP, conforme o PRD.

Epic 1 · Fundação da API

  • Configuração de Docker Compose, banco dev, locale, timezone, auditoria JPA, Flyway, Actuator, CORS e UUID.

Epic 2 · Gestão de Usuários

  • Modelagem da entidade Usuario com UUID v6, auditoria, DTOs, mappers, criação pública e hash de senha seguro.

Epic 3 · Segurança, Autenticação e Autorização

  • JWT, login, claims, consulta de usuário, listagem restrita, alteração de senha, SecurityConfig consolidada e auditoria do usuário autenticado.

Epic 4 · Tratamento de Erros, Documentação e Testes

  • ApiExceptionHandler, payload padronizado, documentação OpenAPI, Swagger UI e testes automatizados dos cenários críticos.

Epic 5 · Onboarding KYC/KYB

  • Onboarding documental, coleta e validação de documentos, verificações de identidade, trilha operacional de aprovação e pendências.

Epic 6 · Análise de Crédito

  • Esteira de análise, parecer operacional, score, decisão, fundamentos e integração ao fluxo de proposta.

Epic 7 · Formalização Contratual

  • Geração de contrato, aceite, assinatura, status de formalização, bloqueio de desembolso e integração futura com assinatura eletrônica.

Epic 8 · Cobrança e Inadimplência

  • Cobrança de parcelas, agenda, status, histórico, acompanhamento de atraso, reprocessos e suporte ao financeiro.

Epic 9 · Backoffice Operacional

  • Fila operacional, visibilidade consolidada, tratamento manual de inconsistências, comentários e organização da operação assistida.

Epic 10 · Jornada da Empresa Credora

  • Onboarding, elegibilidade, governança, aportes, alocações, visão de carteira e acompanhamento das operações financiadas.

Epic 11 · Administração e Governança

  • RBAC evoluído, perfis internos, parametrizações, auditoria administrativa, cadastros mestres e controles operacionais.

Epic 12 · Fundação Frontend

  • Setup do projeto Angular 20 (Standalone Components + Signals + SCSS puro), tradução dos tokens dos dois design systems para variáveis SCSS reutilizáveis, telas públicas seguindo Apple (landing, login, cadastro), shell autenticado seguindo Notion (header, menu lateral, breadcrumbs), biblioteca interna de componentes Notion (botões, inputs, cards, tabelas, modais, toasts, loaders), telas autenticadas iniciais consumindo as APIs das Sprints 1-4 (perfil, alterar senha, admin de usuários, dashboard inicial), guards de rota, controle de sessão, integração HTTP e tratamento padronizado de erros 401/403/404/409. Entrega o "Frontend MVP" navegável e independente das jornadas de negócio.

Epic 13 · Frontend de Jornadas

  • Implementação das telas funcionais das jornadas (tomador, empresa credora, financeiro interno, backoffice e governança avançada) sobre a Fundação Frontend, todas no design system Notion, consumindo as APIs publicadas pelas Epics 5-11. Depende: Epic 12 entregue + APIs das Epics 5-11 publicadas e estáveis.

Epic 14 · Mobile SEP

  • App mobile com baseline Angular 20.x + Ionic 8.4+ + Capacitor 6 (avaliação opcional de upgrade para Angular 21 na fase de implementação se Ionic e plugins Capacitor confirmarem compatibilidade), seguindo o design system Notion adaptado para toque, tabs inferiores e navegação em pilha. Cobre apenas tomador e empresa credora; financeiro, backoffice e administração permanecem no web.

Epic 15 · Movimentação Pix

  • Desembolso, recebimento, conciliação, webhooks, rastreabilidade, idempotência, auditoria e integração operacional do Pix.

Epic 16 · Infraestrutura AWS Futura

  • Planejamento e implementação de ambientes remotos na AWS (EC2 + RDS PostgreSQL), seeds por ambiente, deploy por branch/ambiente, secrets separados, migrations versionadas. Trilha técnica habilitadora; só inicia após a conclusão completa do login, autenticação e autorização.

11. Convenções Obrigatórias

Checklist de padrões que toda task deve respeitar.

IDs · Persistência

  • UUID em todas as entidades
  • Geração via java-uuid-generator 5.1.0, preferência v6
  • Tipo uuid nativo no PostgreSQL
  • java.util.UUID no Java
  • Tabelas e colunas em português
  • Sem soft delete nesta fase

Segurança

  • Senhas com BCryptPasswordEncoder
  • JWT sem refresh token
  • Claims mínimas: sub, email, roles
  • sub carrega o UUID do usuário
  • Logout tratado no cliente (descarta o token)
  • CORS explícito no SecurityConfig

Auditoria

  • JPA Auditing com @EntityListeners(AuditingEntityListener.class)
  • Campos dataCriacao, dataModificacao, criadoPor, modificadoPor
  • AuditorAware<String> retorna UUID do principal
  • Fallback "system" quando não há autenticação

Contratos da API

  • Versionamento em /api/v1
  • DTOs obrigatórios (records Java 21), MapStruct obrigatório (substituiu ModelMapper)
  • Datas em ISO-8601 com offset
  • Termos username, role, accessToken, tokenType podem ficar em inglês
  • Listas simples como array nesta fase, evoluindo para paginação sem quebra
  • ApiExceptionHandler com payload padronizado

Regras das Camadas

  • Controllers só recebem request e devolvem response
  • Regras ficam no service
  • Acesso a banco só no repository
  • Entidades JPA só no model
  • DTOs não são reutilizados como entidade
  • Classes de segurança não carregam regra de domínio

Operacional

  • Flyway aplica migrations no boot
  • Actuator expõe /actuator/health
  • Swagger UI disponível em ambiente de dev
  • Locale pt-BR · Timezone America/Sao_Paulo
  • Parar para teste local manual ao final de cada task
  • Commits pelo agente · push e PR manuais
