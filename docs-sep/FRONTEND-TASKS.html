Trilha de Tasks Frontend Web

Guia consolidado para os desenvolvedores frontend do projeto SEP, contendo somente a trilha web. As tasks abaixo estao em ordem de execucao, com objetivo, entregaveis, pre-requisitos, dependencias, criterios de verificacao e sprint de origem.

5 F-Sprints consolidadas

1 arquivo unico para consulta operacional da squad web

Como usar este arquivo. Leia primeiro a visao geral e a matriz resumida. Depois execute as tasks na ordem indicada. Quando houver paralelismo seguro, ele esta explicitado nas dependencias.
Importante: este arquivo nao lista tasks de backend nem mobile. Ainda assim, algumas tasks frontend dependem de gates externos do backend, como disponibilidade dos endpoints de auth e usuarios.

Ordem recomendada de implementacao web

  1. Base web: scaffold, tooling, testes e CI.
  2. Tokens e showcase dos design systems.
  3. Landing, login e cadastro com mocks aprovados.
  4. Auth real, shell autenticado, guards e interceptors.
  5. Perfil, alterar senha, administracao de usuarios, dashboard e smoke E2E.

Isso segue a sequencia dos specs 100 a 104 e a ordem do WEB-SCREENS-PLAN.md.

F-Sprint 0 - Setup Angular

Objetivo: fundacao tecnica do frontend web
F-0.1 F-Sprint 0 Base Angular

Scaffold Angular 20.x

Inicializar o projeto Angular com Standalone Components, Signals, SCSS e estrutura modular por feature.

O que fazer

  • Criar o projeto em apps/sep-frontend/.
  • Ativar strict mode.
  • Separar core, shared, layout e features.
  • Preparar features/public e features/authenticated.

Pre-requisitos e dependencias

  • Node.js LTS >= 20.x.
  • Sem dependencia interna anterior.

Entregaveis

  • Estrutura inicial do app Angular.
  • Pasta de estilos com base SCSS.
  • Projeto subindo em localhost:4200.

Criterios de verificacao

  • App sobe localmente.
  • Sem NgModule.
  • Estrutura de pastas criada.
F-0.2 F-Sprint 0 Qualidade

Tooling: ESLint + Prettier + Stylelint + Husky + lint-staged

Configurar qualidade de codigo, formatacao e pre-commit hooks do frontend.

O que fazer

  • Configurar eslint.config.js em flat config.
  • Adicionar Prettier e Stylelint.
  • Plugar Husky + lint-staged no pre-commit.

Pre-requisitos e dependencias

  • Depende de F-0.1.

Entregaveis

  • Lint e format padronizados.
  • Hook de pre-commit funcional.
  • Config de lint-staged no package.json.

Criterios de verificacao

  • npm run lint sem erros em codigo valido.
  • Pre-commit bloqueia codigo mal formatado.
  • Auto-fix para ajustes triviais.
F-0.3 F-Sprint 0 Testes

Vitest + Playwright + MSW

Preparar testes unitarios, E2E e mocks de API para desenvolvimento e execucao local.

O que fazer

  • Configurar Vitest.
  • Configurar Playwright com smoke placeholder.
  • Criar estrutura src/mocks/ com MSW.

Pre-requisitos e dependencias

  • Depende de F-0.1 e F-0.2.

Entregaveis

  • vitest.config.ts
  • playwright.config.ts
  • Handlers MSW iniciais
  • e2e/smoke.spec.ts

Criterios de verificacao

  • npm run test executa.
  • npm run e2e executa.
  • MSW intercepta chamadas em dev.
F-0.4 F-Sprint 0 CI

GitHub Actions CI Frontend

Subir pipeline minima de lint, test e build para validar PRs do frontend.

O que fazer

  • Criar .github/workflows/frontend-ci.yml.
  • Executar npm ci, lint, test e build.
  • Restringir paths para apps/sep-frontend/**.

Pre-requisitos e dependencias

  • Depende de F-0.1, F-0.2 e F-0.3.

Entregaveis

  • Workflow de CI do frontend.
  • PR testavel com status checks.

Criterios de verificacao

  • PR quebra quando lint, teste ou build falham.
  • Pipeline idealmente abaixo de 5 minutos.

F-Sprint 3 - Auth real e shell autenticado

Objetivo: sair do mock e conectar o frontend ao backend real
F-3.1 F-Sprint 3 Integracao real

Substituir MSW por API real (login + /me)

Apontar auth para o backend local real, mantendo fallback offline opcional.

O que fazer

  • Tornar URL do backend configuravel por environment.
  • Atualizar auth.service.ts para consumir backend real.
  • Manter MSW via flag dev-offline.

Pre-requisitos e dependencias

  • Depende de gate externo: backend com /auth/login e /auth/me.
  • Backend rodando localmente.

Entregaveis

  • environment.ts e variao offline.
  • MSW condicional.
  • Auth real operacional.

Criterios de verificacao

  • Login real funciona.
  • JWT enviado em Authorization: Bearer.
  • Fallback offline disponivel.

F-3.3 F-Sprint 3 Seguranca de rotas

Guards funcionais + interceptors

Proteger rotas, anexar token nas requests e tratar 401/403.

O que fazer

  • Criar auth.guard e role.guard.
  • Criar interceptors de auth e erro.
  • Adicionar pagina de acesso negado.

Pre-requisitos e dependencias

  • Depende de F-3.1 e F-3.2.

Entregaveis

  • Guards funcionais Angular 20.
  • Interceptors HTTP.
  • Pagina /access-denied.

Criterios de verificacao

  • Rota protegida sem token redireciona para login.
  • 401 limpa sessao.
  • 403 mostra acesso negado.

F-4.5 F-Sprint 4 E2E

Smoke E2E Playwright

Cobrir o golden path do frontend MVP: register, login, shell, perfil, senha e admin.

O que fazer

  • Criar golden-path.spec.ts.
  • Criar admin-flow.spec.ts.
  • Preparar fixtures de usuarios.

Pre-requisitos e dependencias

  • Depende de F-4.1, F-4.2, F-4.3 e F-4.4.
  • Gate externo: backend completo e estavel para smoke.

Entregaveis

  • Suite E2E Playwright do frontend MVP.
  • Fixtures de teste.

Criterios de verificacao

  • E2E roda contra backend local.
  • Screenshots de falha disponiveis.
  • Tempo alvo abaixo de 3 minutos.

Gates externos relevantes para o frontend web

  • F-3.1 depende de backend com POST /api/v1/auth/login e GET /api/v1/auth/me.
  • F-4.2 depende de backend com PATCH /api/v1/usuarios/{id}/senha.
  • F-4.3 depende de backend com GET /api/v1/usuarios e GET /api/v1/usuarios/{id}, alem de autorizacao por perfil.
  • F-4.5 depende do backend local rodando de forma estavel para execucao de smoke E2E.

Esses gates aparecem aqui apenas como condicionantes operacionais da trilha web. Eles nao transformam este documento em um plano de backend.
